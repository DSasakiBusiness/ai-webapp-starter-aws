# Run AI Evals

## この skill を使う場面

- AI 機能の品質をリリース前に確認する場合
- プロンプト変更後の品質回帰を検証する場合
- LLM モデル変更の影響を評価する場合

## 入力前提

- 評価対象の AI 機能が動作可能であること
- ゴールデンデータセット（入力と期待出力のペア）
- 評価基準の定義

## 実行手順

### Step 1: 評価データセットの準備

```json
// tests/ai-evals/datasets/[feature].json
{
  "name": "feature-eval",
  "version": "1.0",
  "cases": [
    {
      "id": "EVAL-001",
      "input": { "query": "テスト入力" },
      "expected": {
        "format": "json",
        "schema": { "type": "object", "required": ["answer", "sources"] },
        "constraints": ["answer は 500 文字以内", "sources は 1 件以上"]
      }
    }
  ]
}
```

### Step 2: 評価スクリプトの実行

```bash
npx ts-node tests/ai-evals/run-evals.ts --dataset=[feature]
```

### Step 3: 評価観点

- **構造評価**: 出力が期待する JSON Schema に準拠しているか
- **長さ制約**: 出力が指定された文字数/トークン数の範囲内か
- **引用正確性**: RAG の場合、ソース引用が正確か
- **失敗時動作**: エラー入力に対して適切なフォールバックが返るか
- **境界条件**: 空入力、長大入力、特殊文字に対して安全に処理するか
- **コスト**: トークン使用量が想定範囲内か

### Step 4: 結果記録

評価結果を JSON で出力し、前回との差分を確認する。合格基準を設定し、CI で自動判定する。

## 判断ルール

- 全評価ケースで構造評価が合格 → リリース可能
- 構造評価の合格率が 90% 未満 → プロンプト改善が必要
- コストが前回比 150% 以上 → プロンプトの最適化を検討

## 出力形式

評価結果の JSON と、前回比較のサマリーレポート。

## 注意点

- 評価には実際の LLM API を使うため、コストが発生する
- 評価データセットは定期的に更新する
- 厳密な文字列一致ではなく構造・制約ベースで評価する

## 失敗時の扱い

- LLM API エラー: 再試行し、3 回失敗したら評価をスキップして記録
- 評価基準未定義: clarify-ai-requirements で基準を定義してから再実行

# Clarify Product Requirements

## この skill を使う場面

- 新機能の開発を開始する前に要件が不明確な場合
- ステークホルダーから受け取った要件が抽象的で実装に落とし込めない場合
- 既存機能の改修で影響範囲が不明な場合

## 入力前提

- ステークホルダーからの初期要件（口頭、テキスト、画面イメージ等）
- 対象ユーザーの情報
- 既存システムの制約（ある場合）

## 実行手順

### Step 1: 要件の分解

受け取った要件を以下の観点で分解する:

1. **ユーザーストーリー化**: 「As a [ユーザー], I want [機能], so that [価値]」形式に変換する
2. **機能要件の列挙**: 具体的に何ができるべきかを箇条書きにする
3. **非機能要件の確認**: 性能、セキュリティ、可用性の要件を確認する

### Step 2: 受け入れ条件の定義

各ユーザーストーリーに対して Given-When-Then 形式で受け入れ条件を定義する:

```
Given: [前提条件]
When: [操作]
Then: [期待結果]
```

### Step 3: 不明点の洗い出し

以下の観点で不明点を列挙する:

- ユーザーフローの分岐（正常系 / 異常系 / 境界条件）
- データの入出力形式
- 既存機能との関係（依存、競合、共有データ）
- 権限・認可の要件
- AI 機能が含まれる場合の精度要件

### Step 4: 優先度付け

MoSCoW 法で分類する:

- **Must**: リリースに必須
- **Should**: 強く望ましい
- **Could**: あれば良い
- **Won't**: 今回は対象外

### Step 5: 合意形成

整理した仕様をステークホルダーに提示し、合意を得る。

## 判断ルール

- 不明点が 3 つ以上ある場合: ステークホルダーとの追加ヒアリングを提案する
- AI 機能を含む場合: clarify-ai-requirements skill も併用する
- テスト範囲が不明な場合: clarify-test-scope skill も併用する

## 出力形式

```markdown
# [機能名] 要件定義

## ユーザーストーリー
- US-001: As a [ユーザー], I want [機能], so that [価値]

## 受け入れ条件
### US-001
- Given: [前提]
  When: [操作]
  Then: [結果]

## 非機能要件
- 性能: [具体値]
- セキュリティ: [要件]

## 優先度
- Must: [項目]
- Should: [項目]
- Could: [項目]

## 未決定事項
- [不明点と確認先]
```

## 注意点

- 要件は実装者が読んで着手できる粒度にする
- 「〜できること」のような曖昧な記述を避け、具体的な入出力を記述する
- AI 機能の精度は「完璧」ではなく、具体的な数値目標を設定する

## 失敗時の扱い

- ステークホルダーが不在で確認できない場合: 既知情報で仮定義し、仮定条件を明示する
- 要件が矛盾する場合: 矛盾点を明示し、判断を仰ぐ
- 技術的に実現困難な要件: solution-architect に技術検証を依頼する

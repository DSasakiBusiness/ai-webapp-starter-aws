# Review Security with PentAGI

## この skill を使う場面

- リリース前のセキュリティ検証を実施する場合
- 新しい API エンドポイントのセキュリティを確認する場合
- LLM 特有の攻撃面を評価する場合

## 入力前提

- ステージングまたは隔離環境がデプロイ済みであること
- PentAGI がセットアップ済みであること（setup-pentagi-scan 参照）
- スキャン対象の承認が得られていること

## ⚠️ 安全上の絶対ルール

1. **本番環境に対して PentAGI を絶対に実行しない**
2. ステージングまたは完全に隔離された環境でのみ実行する
3. スキャン対象は事前に明示的な承認を得る
4. 結果は必ず人間が再確認する
5. 誤検知を前提として扱う
6. スキャン前に対象環境の復元手順を確認する

## 実行手順

### Step 1: 対象環境の確認

```bash
# 対象 URL がステージングであることを確認
echo $PENTAGI_TARGET_URL
# 本番 URL との一致がないことを二重確認
```

### Step 2: スキャン範囲の定義

検証対象:
- API エンドポイント（認証・認可）
- 入力バリデーション（SQLi, XSS, SSRF）
- ファイルアップロード
- セッション管理（JWT）
- CORS 設定
- LLM 固有の攻撃面（プロンプトインジェクション、出力漏洩）

### Step 3: PentAGI スキャン実行

PentAGI をステージング環境に対して実行する。実行時には隔離 Docker ネットワーク内で行う。

### Step 4: 結果分析

検出された脆弱性を以下の形式で分類する:

| ID | 重大度 | 攻撃ベクター | 影響範囲 | 誤検知の可能性 |
|---|---|---|---|---|
| VULN-001 | Critical | SQLi | データ漏洩 | 低 |

### Step 5: 手動再検証

PentAGI の結果を手動で再検証する（誤検知を前提）:
1. 各脆弱性の再現手順を実行する
2. 再現できたものは「確認済み」とマークする
3. 再現できないものは「誤検知の可能性」とマークする

### Step 6: 修正提案

確認済みの脆弱性に対して:
- 根本原因の分析
- 修正方法の提案（コード例）
- 修正後の再検証手順

## 判断ルール

- Critical: 即座に修正、リリースブロック
- High: リリース前に修正必須
- Medium: 次スプリントで対応
- Low: バックログに追加

## 出力形式

```markdown
# セキュリティレビュー結果

## 概要
- スキャン日時: [日時]
- 対象環境: [ステージング URL]
- スキャン範囲: [範囲]

## 検出結果サマリー
- Critical: [件数]
- High: [件数]
- Medium: [件数]
- Low: [件数]
- 誤検知: [件数]

## 詳細
### VULN-001: [脆弱性名]
- 重大度: [Critical/High/Medium/Low]
- ステータス: [確認済み/誤検知の可能性]
- 攻撃ベクター: [詳細]
- 再現手順: [手順]
- 修正提案: [コード例]
- 再検証手順: [手順]
```

## 注意点

- PentAGI の結果を鵜呑みにしない（必ず手動再検証する）
- スキャンがサービスに負荷を与える可能性があるため、事前にリソースを確認する
- LLM 攻撃面のテストは少量のプロンプトから開始し、段階的に拡大する

## 失敗時の扱い

- PentAGI が起動しない: Docker ログを確認し、設定を見直す
- 誤検知が多い: 除外ルールを定義し、PentAGI の設定を調整する
- ステージング環境が利用不可: 手動セキュリティチェックリストに切り替える

# TDD Feature Delivery

## この skill を使う場面

- 新機能を TDD で開発する場合
- 既存機能を TDD で改修する場合
- tdd-coach の指示に従って開発を進める場合

## 入力前提

- product-manager の受け入れ条件（Given-When-Then）
- tdd-coach が承認したテスト方針
- 対象のテスト種類（unit / integration）

## 実行手順

### Step 1: Red（失敗するテストを書く）

1. 受け入れ条件を 1 つ選ぶ
2. テストファイルを作成する
3. テストを書く（この時点で実装は存在しない）
4. テストを実行して失敗することを確認する

```bash
# テスト実行
npm run test:unit -- --watch
# Docker 経由
docker compose exec api npm run test:unit -- --watch
```

テスト命名規則:
```typescript
describe('[対象クラス/関数]', () => {
  describe('[メソッド名]', () => {
    it('should [期待動作] when [条件]', () => {
      // Arrange
      // Act
      // Assert
    });
  });
});
```

### Step 2: Green（最小限の実装でテストを通す）

1. テストを通すための最小限のコードを書く
2. テストが通ることを確認する
3. この段階では美しいコードは不要、テストが通ればよい

### Step 3: Refactor（テストが通る状態でリファクタリング）

1. コードの重複を排除する
2. 命名を改善する
3. 構造を整理する
4. テストが引き続き通ることを確認する

### Step 4: 次の受け入れ条件に進む

Step 1-3 を全ての受け入れ条件に対して繰り返す。

### Step 5: 統合テスト（該当する場合）

ユニットテストが全て通った後、実際の DB を使った統合テストを追加する。

## 判断ルール

- テストが 5 秒以上かかる → テストの粒度を再検討する
- 実装が複雑になりすぎる → 設計を見直す（solution-architect に相談）
- テストが実装の内部構造に依存 → 振る舞いベースに書き換える
- AI 機能のテスト → 構造評価、失敗時動作、境界条件で検証する

## 出力形式

- テストファイル（`.spec.ts` または `.test.tsx`）
- 実装ファイル
- リファクタリング後の最終コード

## 注意点

- 実装前にテストを必ず書く（テストなしの実装は禁止）
- 1 サイクルで 1 つの受け入れ条件に集中する
- テスト間の依存を排除する（各テストは独立して実行可能）
- AI 機能のテストで厳密な文字列一致を使わない

## 失敗時の扱い

- テストが書けない → 受け入れ条件が不明確。product-manager に差し戻す
- テストがflaky → 環境依存、タイミング依存を排除する
- リファクタリングでテストが壊れる → リファクタリングの粒度が大きすぎる。小さいステップに分ける

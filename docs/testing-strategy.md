# テスト戦略

## テスティングピラミッド

```
        /‾‾‾‾‾‾‾‾\
       /   E2E    \        少数・高コスト・ユーザーフロー検証
      /─────────────\
     / Integration   \     中程度・DB含む・モジュール間連携
    /─────────────────\
   /    Unit Tests     \   大量・高速・ロジック単体検証
  /─────────────────────\
```

## 種類別方針

| 種類 | フレームワーク | カバレッジ目標 | 実行頻度 |
|---|---|---|---|
| Unit | Jest | 80% line, 70% branch | 毎コミット |
| Integration | Jest + Prisma | 主要フロー網羅 | 毎 PR |
| E2E | Playwright | 主要ユーザーフロー | 毎 PR |

## TDD サイクル

1. **Red**: 失敗するテストを先に書く
2. **Green**: 最小限の実装で通す
3. **Refactor**: テストが通る状態で改善

## AI 機能のテスト

- 厳密文字列一致は使わない
- 構造評価（JSON Schema 準拠）を使う
- 失敗時動作（フォールバック）を検証する
- 境界条件（空入力、長大入力）を検証する
- トークン使用量を確認する
